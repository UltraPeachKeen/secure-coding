## 1. 프로젝트 개요

본 프로젝트는 Flask 기반의 작은 중고거래 플랫폼을 직접 분석·설계·구현·테스트·유지보수하며 시큐어 코딩을 개발 생명주기 전체에 적용한 결과물이다. 단순히 기능이 동작하는 데서 그치지 않고 인증과 인가, 입력 검증, 데이터 무결성, 안전한 파일 처리와 오류 처리까지 각 단계의 기본 조건으로 삼았다.

사용자는 가입 후 상품을 등록하고 검색할 수 있으며 전체 또는 1:1 채팅으로 소통한다. 악성 사용자와 상품은 사유를 적어 신고할 수 있고 서로 다른 사용자 신고가 기본 3회 누적되면 사용자 휴면 또는 상품 차단이 자동 적용된다. 송금은 실제 금융 서비스가 아닌 사이트 내부의 가상 잔액 이동으로 구현하였다. 관리자는 사용자·상품·신고·거래와 자신의 관리 작업 기록을 조회·관리한다.

## 2. 요구사항 분석

### 2.1 기능 요구사항

강의 슬라이드 마지막 최소 요구사항을 기준으로 가입, 상품 등록과 조회, 사용자 소통, 악성 사용자·상품 차단, 사용자 간 송금, 상품 검색, 관리자 전체 관리를 필수 기능으로 선정했다. 슬라이드의 상세 시스템 설계 예시에 나온 로그인·프로필·마이페이지·전체/1:1 채팅·신고 사유·신고 누적 제재·내 상품 관리도 포함하였다.

구현 기능은 회원가입·로그인·로그아웃, 공개 프로필, 소개글·비밀번호 변경, 상품 CRUD와 사진, 검색과 내 상품, 전체/1:1 채팅, 사용자/상품 신고와 중복 방지, 자동 차단/휴면, 가상 송금과 내역, 관리자 사용자·상품·신고·거래·감사 로그이다.

### 2.2 비기능 요구사항

보안, 데이터 무결성, 재현성, 유지보수성, 시험 가능성, 사용성을 비기능 요구사항으로 정했다. 다른 컴퓨터에서 README만으로 실행할 수 있도록 Docker Compose와 로컬 가상환경 절차를 모두 제공했다. 기능별 Blueprint와 자동 테스트를 사용해 변경의 영향 범위를 줄였다.

### 2.3 요구사항 해석과 변경

슬라이드의 제출일 7월 23일은 디스코드 공지에서 오기로 정정되었으므로 7월 24일을 기준으로 했다. 신고의 “일정 횟수”는 명시되지 않아 기본 3회로 정하고 환경변수로 조정 가능하게 했다. 실시간 채팅은 과제 규모와 안정성을 고려해 3초 AJAX polling으로 구현했다. “누구나 상품을 볼 수 있음”은 비로그인 사용자도 활성 상품 목록과 상세를 볼 수 있다는 의미로 해석했다.

## 3. 시스템 설계

### 3.3 데이터베이스 구조

`users`는 계정명, 비밀번호 해시, 소개, 가상 잔액, 역할, 상태, 신고 수를 저장한다. `products`는 판매자 FK, 상품명, 설명, 가격, UUID 이미지명, 상태와 신고 수를 저장한다. `chat_messages`는 발신자와 선택적 수신자를 가지며 수신자가 NULL이면 전체 채팅, 값이 있으면 1:1 채팅이다.

`reports`는 신고자와 사용자 또는 상품 중 정확히 하나의 대상을 저장한다. CHECK 제약으로 대상 하나만 존재하게 하고, 신고자-대상 복합 UNIQUE로 중복 신고를 DB에서도 막는다. `transfers`는 발신자·수신자·양수 금액과 시각을 저장한다. `audit_logs`는 관리자의 상태 변경 작업을 남긴다.

주요 관계는 사용자 1:N 상품, 사용자 1:N 발신/수신 메시지, 사용자 1:N 신고, 상품 1:N 신고, 사용자 1:N 발신/수신 송금이다. SQLite 연결마다 foreign key pragma를 활성화했다. 잔액·가격·금액·상태에는 CHECK 제약을 두었다.

### 3.4 권한 설계

비회원은 활성 상품과 프로필만 조회한다. 활성 일반 사용자는 자신의 상품만 수정·삭제하며 채팅·신고·송금을 사용할 수 있다. 휴면 사용자는 로그인과 쓰기 기능이 제한된다. 관리 기능은 화면의 링크를 숨기는 것과 별개로 모든 `/admin` 라우트에서 역할과 활성 상태를 검사한다. 관리자는 관리 화면에서 사용자와 상품 상태, 신고 처리 상태를 바꾸고 거래와 감사 기록을 조회한다.

## 4. 구현 내용

### 4.1 회원

회원가입은 아이디 3~32자의 영문·숫자·밑줄과 비밀번호 10~128자를 검증한다. 아이디는 모델의 UNIQUE와 폼 사전 검사로 중복을 방지한다. 비밀번호는 Werkzeug의 scrypt 방식으로 salted hash만 저장한다. 로그인 오류는 존재 여부를 구분하지 않는 문장으로 통일하고 휴면 계정은 로그인시키지 않는다. 비밀번호 변경은 현재 비밀번호 확인 후 처리하고 즉시 로그아웃시킨다.

### 4.2 상품과 업로드

상품명, 설명, 가격을 서버에서 검증하고 가격은 1원 이상 10억 원 이하 정수로 제한했다. 검색은 SQLAlchemy 바인딩 질의로 제목과 설명을 조회하고 LIKE 메타문자를 escape한다. 수정과 삭제는 판매자 ID가 현재 사용자 ID와 같은지 서버에서 검사한다.

업로드는 요청 전체 2MB 제한, 원본 확장자 허용 목록, Pillow 실제 이미지 디코딩, PNG/JPEG/GIF/WebP 포맷과 5000x5000 이하 크기를 확인한다. UUID 파일명을 생성하고 다시 인코딩해 원본 파일명, 경로 조작 문자열과 불필요한 원본 구조를 보존하지 않는다.

### 4.3 채팅

전체 채팅은 `receiver_id IS NULL`, 1:1 채팅은 두 사용자 방향의 조건으로 조회한다. API는 로그인 사용자가 속한 대화만 반환한다. 최근 100개를 기본 표시하고 3초마다 마지막 ID 이후의 메시지만 조회한다. 메시지는 최대 500자로 제한한다.

### 4.4 신고와 자동 제재

자기 자신, 관리자, 자기 상품은 신고할 수 없다. 신고자와 사용자/상품 대상의 UNIQUE 제약이 동시에 도착한 중복 요청도 막는다. 새 신고가 저장될 때 기각되지 않은 신고 수를 다시 계산하며 임계치 이상이면 상품을 `blocked`, 사용자를 `dormant`로 변경한다. 관리자는 신고를 승인 또는 기각하고 대상 상태를 별도로 복구할 수 있다.

### 4.5 가상 송금

받는 사용자가 존재하고 활성 상태인지, 자기 자신이 아닌지, 금액이 양수 범위인지 확인한다. 차감은 `WHERE id = sender AND balance >= amount` 조건부 UPDATE로 수행한다. 한 행이 변경된 경우에만 수신자 증가와 거래 이력을 같은 세션에서 처리하고 한 번 commit한다. 오류가 나면 rollback하여 부분 반영을 막는다.

### 4.6 관리자

관리자는 사용자 활성/휴면, 상품 활성/차단, 신고 승인/기각을 처리한다. 전체 거래를 읽고 상태 변경은 `audit_logs`에 관리자, 작업, 대상, 상세와 시각으로 남긴다. 자기 관리자 계정을 스스로 휴면 처리하는 작업은 거부한다.

## 5. 적용한 보안 통제와 수정 전후

첫째, 평문 비밀번호 대신 scrypt hash를 사용했다. 둘째, 문자열 SQL 대신 SQLAlchemy 바인딩 질의를 사용했다. 셋째, Jinja 자동 이스케이프와 DOM `textContent`, CSP로 저장형 및 DOM XSS를 방어했다. 넷째, Flask-WTF 전역 CSRF와 POST 상태 변경을 사용했다. 다섯째, 모든 객체 작업에서 로그인·활성·소유권·관리자 역할을 서버에서 확인했다.

여섯째, 업로드 확장자만 믿지 않고 실제 이미지 검증·크기 제한·UUID명·재인코딩을 적용했다. 일곱째, 송금은 조건부 차감과 트랜잭션으로 금액 변조와 부분 반영을 막았다. 여덟째, 중복 신고는 사전 검사만이 아니라 DB UNIQUE로 막았다. 아홉째, SECRET_KEY가 없으면 운영 앱이 시작되지 않으며 `.env`, DB와 업로드는 Git에서 제외했다. 열째, 내부 오류를 일반 화면으로 바꾸고 CSP, nosniff, frame, referrer, permissions 보안 헤더를 적용했다.

각 항목의 위험한 형태, 수정, 검증은 저장소의 `docs/security-changes.md`에 정리하였다.

## 7. 유지보수 과정

첫 번째로 신고가 상품을 FK로 참조하는 상황에서 물리 삭제가 감사 기록을 없애거나 무결성 오류를 만들 수 있음을 발견했다. 삭제 요구사항의 사용자 경험은 유지하면서 DB에는 `deleted` 상태를 남기는 소프트 삭제로 변경했고 공개 목록과 이미지 제공을 차단했다.

두 번째로 여러 신고자 누적 자동 테스트에서 내부 세션 값을 강제로 교체한 테스트 방식이 실제 사용자 전환을 정확히 재현하지 못했다. 실제 로그아웃/로그인 요청으로 테스트를 변경하고 3인의 신고가 각각 저장되어 자동 차단·휴면되는 것을 재확인했다.

세 번째로 개발 편의를 위한 고정 SECRET_KEY 기본값이 배포 실수로 이어질 수 있어, 비테스트 앱은 환경변수가 없으면 시작 단계에서 실패하도록 변경했다. 네 번째로 검색의 LIKE 메타문자가 예상보다 넓은 결과를 만들 수 있어 `%`, `_`, 역슬래시를 escape했다. 다섯 번째로 한글 상위 폴더에서 Docker Compose가 자동 프로젝트명을 만들지 못하는 문제를 재현해 Compose 파일에 `tiny-market` 이름을 명시했다.

## 8. 재현성과 실행

사용자는 `.env.example`을 `.env`로 복사하고 무작위 SECRET_KEY를 설정한 뒤 `docker compose up --build`로 실행한다. 컨테이너는 DB 테이블을 초기화하고 Gunicorn을 실행한다. `docker compose exec web flask --app run.py seed-demo`로 관리자와 일반 테스트 계정을 생성할 수 있다. Docker가 없으면 WSL Ubuntu에서 Python 가상환경, requirements 설치, DB 초기화 순서로 실행한다. 전체 절차는 README에 명시하였다.

## 9. AI 도구 활용

과제에서 AI 도구 활용이 권장되어 요구사항 분류, DB/URL/권한 설계 초안, Flask 코드와 테스트 케이스 생성, 오류 원인 분석, 보안 검토, README와 보고서 초안 작성에 Codex를 활용했다. 생성 결과를 그대로 제출하지 않고 첨부 자료와 대조하고 코드를 직접 실행했다. 실패한 테스트와 설계상 삭제 무결성 문제를 분석해 수정한 뒤 전체 테스트를 반복하였다.

## 10. 제한사항과 향후 개선

본 시스템의 잔액은 실제 자산이 아니며 금융 서비스로 사용할 수 없다. SQLite와 단일 프로세스는 과제 규모를 위한 선택으로, 대규모 서비스는 PostgreSQL과 더 강한 동시성·격리 설계가 필요하다. polling은 WebSocket보다 지연과 요청이 많다. 공개 인터넷 배포 전에는 Redis 기반 로그인/요청 rate limiting, HTTPS 프록시와 Secure cookie, DB migration, 백업/복구, 악성 이미지 스캔과 모니터링을 추가해야 한다.

## 11. 결론

최소 요구사항을 기능 단위로만 구현하지 않고 요구사항 추적, 서버 측 권한, DB 제약, 트랜잭션, 업로드 검증과 자동/수동 검증까지 연결하였다. 개발 중 발견한 무결성·배포 설정·테스트 방식 문제를 수정하고 회귀 테스트를 통과함으로써 시큐어 코딩은 마지막 보안 점검이 아니라 요구사항과 설계부터 유지보수까지 계속 적용되어야 한다는 점을 확인했다.
